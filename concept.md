# Конструктор аналитических выборок (семантический поиск + Excel-подобная таблица)

## Обновлённая спецификация

## 1. Введение

Этот документ описывает концепцию и архитектуру веб-сервиса для интеллектуального отбора бюджетных данных с использованием **семантического поиска и контекстного анализа** (без LLM/чат-бота, без генерации SQL и без передачи данных в модель).

Сервис позволяет:

- Формировать аналитические выборки по запросу на естественном языке.
- Дорабатывать результат вручную через фильтры и настройки таблицы.
- Управлять таблицей через запрос (семантика → DSL) и вручную.
- Обеспечивать воспроизводимость результата (привязка к `dataset_version`) и устойчивость: при сбое семантического слоя система не падает, а деградирует до ручного режима.

Документ ориентирован на прототип хакатона, но сохраняет архитектурные принципы «backend — единственный доверенный слой».

## 2. Цели продукта и ключевые фичи

### Основные цели

- Упростить получение аналитических выборок по бюджетным данным «в один запрос».
- Сохранить прозрачность и контроль для аналитиков, которые привыкли к ручным фильтрам и Excel.
- Обеспечить неизменяемость исходных данных и воспроизводимость результатов.
- Минимизировать риски отказа сервиса из-за тяжёлых SQL-запросов и ошибок состояния.

### Ключевые фичи

#### Умный интерфейс для неподготовленного пользователя

- «Google-подобный» экран с одной строкой поиска и готовыми шаблонами.
- Подсказки по бюджетам, получателям, периодам, кодам КЦСР/КФСР/КВР/КОСГУ.
- Всплывающие подсказки с расшифровкой аббревиатур и кодов.

#### Продвинутая бизнес-аналитика

- Автоматический расчёт процента освоения бюджета.
- Цветовая индикация проблемных участков.
- Сравнение периодов, агрегаты, отклонения.

#### Технически корректный backend

- Нормализация CSV в PostgreSQL.
- Асинхронная обработка тяжёлых операций.
- Кэширование повторяющихся выборок (опционально).
- Строгая валидация DSL и отказ от `eval` / динамического исполнения формул.

#### Сильная презентационная часть

- Экспорт в Excel и CSV (PDF — опционально).
- Поддержка планшетов.
- Пресеты и сохранённые сценарии анализа.

## 3. Пользовательские роли

Для прототипа **одна роль**.

### Аналитик

- Авторизуется по логину/паролю.
- Формирует выборки через запрос и/или ручные фильтры.
- Управляет видом таблицы (добавить/убрать колонки, порядок, закрепление).
- Экспортирует отчёты.

## 4. Пользовательские потоки

### 4.1. Авторизация

1. Пользователь открывает сайт.
2. Видит форму логина (логин и пароль).
3. Вводит их.
4. Backend проверяет и создаёт server-side session.
5. Пользователь попадает на главный экран поиска.

#### Правила

- Используются server-side sessions (а не stateless JWT).
- `session_id` хранится в `HttpOnly` cookie.
- TTL сессии — 30 минут, таймаут неактивности — 15 минут.
- Logout завершает сессию.

### 4.2. Главный экран поиска

Экран содержит:

- Логотип.
- Большую строку поиска.
- Кнопку «Ручное фильтрование».

#### Поток по запросу

1. Пользователь вводит запрос (или диктует).
2. Frontend отправляет запрос на backend.
3. Backend формирует контекст: доступные источники/поля/relations, словари значений.
4. Семантический слой возвращает кандидатный DSL-план.
5. Backend валидирует план, применяет ограничения и строит выборку.
6. План сохраняется на сервере и получает `plan_id`.

Если запрос не удалось интерпретировать:

- выводим контролируемую ошибку
- предлагаем ручной режим

#### Поток ручного режима

1. Пользователь нажимает «Ручное фильтрование».
2. Backend создаёт пустой серверный план с обязательными ограничениями.
3. Пользователь переходит к экрану таблицы.

### 4.3. Экран таблицы

#### Разметка

- Слева — таблица.
- Справа сверху — строка «Запрос» (текст/голос) и история запросов.
- Справа снизу — фильтры, настройки и список колонок.

#### Функции

- Ручная настройка выборки.
- Правка через запрос (семантика).
- Экспорт в популярные форматы (xlsx, csv).
- Сохранение пресетов.
- Undo/redo и история версий плана.

#### Ключевое правило

Frontend передаёт только `plan_id`, `plan_version` и действие пользователя. Полный план хранится и модифицируется только на сервере.

#### Вид таблицы (как Excel)

Таблица должна выглядеть просто и знакомо:

- фильтры и сортировка в шапке колонок
- изменение ширины колонок
- закрепление колонок
- горизонтальная прокрутка
- номер строки
- поиск по отображаемым данным
- условное форматирование (например, по % освоения)

При формировании запроса пользователю показываются **именно те данные, которые он хотел** (колонки/метрики/фильтры из запроса). После этого он может убрать/добавить колонки и сохранить пресет.

## 5. Семантический слой и ограничения

### 5.1. Назначение семантического слоя

Семантический слой интерпретирует текст запроса и превращает его в кандидатный DSL-план. Он работает с:

- метаданными источников (какие поля существуют),
- словарями значений (коды/наименования),
- whitelist relations для join по КЦСР и по ключам внутри ГЗ.

### 5.2. Компоненты семантического слоя

- Словарь синонимов полей/сущностей.
- Индекс значений (fuzzy/semantic).
- Intent parser: период, коды, пороги, метрики, колонки, сортировки, агрегаты.
- DSL builder: сборка плана только из whitelist.

### 5.3. Degraded mode

Если семантический слой недоступен или не смог интерпретировать запрос:

- Система не падает.
- Пользователь переводится в режим «ручные фильтры и колонки».
- Уже созданные планы продолжают работать.

## 6. DSL и план выполнения

### 6.1. Общий принцип

Семантический слой не возвращает SQL и не исполняет запросы напрямую. Он возвращает кандидатный DSL-план, который:

- валидируется по JSON Schema / Pydantic,
- нормализуется,
- дополняется серверными ограничениями,
- превращается в безопасный запрос.

### 6.2. Формат DSL

Пример:

```json
{
  "sources": [
    { "name": "rchb", "alias": "r" },
    { "name": "buau", "alias": "b" }
  ],
  "joins": [
    {
      "left": "r",
      "right": "b",
      "relation": "rchb_to_buau_by_kcsr_norm"
    }
  ],
  "filter_tree": {
    "op": "and",
    "items": [
      { "field": "r.kcsr_norm", "cmp": "=", "value": "0220197003" },
      { "field": "r.posting_date", "cmp": "between", "value": ["2025-08-01", "2025-08-31"] }
    ]
  },
  "columns": [
    "r.budget_name",
    "r.kcsr_raw",
    "r.kcsr_norm",
    "r.spend_amount",
    "r.limit_amount",
    "r.execution_percent",
    "b.org_name",
    "b.spend_amount"
  ],
  "operations": [
    { "type": "sort", "by": "r.execution_percent", "direction": "asc" }
  ],
  "limit": 200,
  "offset": 0
}
```

### 6.3. Исправление JOIN-дыры

JOIN больше не может быть неявным.

Вводится обязательный раздел `joins`:

- `relation` не задаётся свободным текстом.
- Вместо этого используется whitelist-справочник отношений, заранее определённый backend.

#### Правила

- Семантический слой может выбирать только `relation` из whitelist.
- Backend сам знает реальные поля соединения.
- Нельзя соединять источники напрямую «по догадке».

### 6.4. Формулы и calculate

Поле `formula` удаляется полностью.

Операция `calculate` допускает только ограниченный набор арифметических действий:

- `add`
- `subtract`
- `multiply`
- `divide`
- `ratio`
- `percent_change`

Все аргументы:

- Должны быть колонками из белого списка.
- Проходят серверную проверку типов.
- Вычисляются сервером через подготовленные SQL-выражения.

`eval`, `exec`, `DataFrame.query`, `DataFrame.eval` и любое выполнение строковых формул запрещены.

### 6.5. AND/OR логика

Плоский список фильтров заменён на `filter_tree`:

- Поддерживаются `and`, `or`, `not`.
- Допускается вложенность.
- Backend ограничивает максимальную глубину дерева, например до 4 уровней.

### 6.6. Надёжность интерпретации запроса

Проблема: запрос может быть неполным, синонимичным или двусмысленным.

Pipeline:

1. Парсинг намерений (период, сущности, метрики, колонки).
2. Матчинг сущностей по словарям/индексу.
3. Генерация DSL только из whitelist.
4. Валидация через Pydantic / JSON Schema.
5. Если интерпретация не удалась — контролируемое сообщение и ручной режим.

### 6.7. Output sanitization

Даже после успешной интерпретации:

- Все строковые значения очищаются и нормализуются.
- Запрещаются управляющие символы.
- Применяется Unicode normalization.
- Значения фильтров проходят параметризацию при генерации SQL.

## 7. Состояние плана, версии и история

### 7.1. Хранение состояния

План хранится на бэкенде как структура:

- `plan_id`
- `plan_version`
- `session_id`
- `dataset_version`
- `current_plan`
- `history[]`
- `created_at`
- `updated_at`

Frontend не хранит authoritative version плана.

### 7.2. Race condition и plan_version

Для исключения гонок:

- Каждое изменение идёт с `plan_id` и `plan_version`.
- Backend применяет optimistic locking.
- Если версия устарела, сервер возвращает `409 Conflict`.
- Фронтенд перезагружает актуальный план и предлагает повторить действие.

### 7.3. Undo/redo

История версий плана обязательна:

- Каждая успешная модификация создаёт `plan_v+1`.
- Хранится минимум последние N версий (для хакатона N=10).
- Доступны действия: `undo`, `redo`, `restore plan_vN`.

## 8. ETL, данные и версионность наборов

### 8.1. Загрузка CSV

Добавляется явный слой ETL:

- Загрузка/подхват CSV/XLSX в staging-зону.
- ETL-процесс валидирует формат.
- Данные нормализуются.
- Затем грузятся в целевые таблицы PostgreSQL.

#### Этапы

1. Upload в `raw_storage`.
2. Проверка схемы файла.
3. Импорт в `staging_*`.
4. Нормализация и дедупликация.
5. Публикация в `mart_*` / `views`.

#### Источники хакатона и связь по КЦСР

В рамках хакатона данные берутся из папок:

- `1. РЧБ/` (CSV)
- `2. Соглашения/` (CSV)
- `3. ГЗ/` (CSV: бюджетные строки / контракты и договора / платежки)
- `4. Выгрузка БУАУ/` (CSV)
- `СКК - Контрольный пример выгрузки/` (эталонный пример вида таблицы)

Все источники связаны по **КЦСР**. Для этого в ETL вводится `kcsr_norm` и whitelist relations (см. `/opt/hakaton/semantic_layer_dictionary.md`).

### 8.2. Версионность данных

Каждая загрузка получает:

- `dataset_version`
- `loaded_at`
- `source_file`
- `checksum`

Планы и выгрузки должны быть привязаны к `dataset_version`, чтобы можно было воспроизвести результат.

### 8.3. Обновления и инвалидирование

При публикации новой версии данных:

- Кэш выборок инвалидируется.
- Экспорты содержат ссылку на `dataset_version`.

## 9. Backend и инфраструктура

### 9.1. Общая схема

- Frontend: Vue.js SPA.
- Backend: Python FastAPI/Django.
- PostgreSQL: primary OLTP/analytics store.
- Redis: кэш, очередь, server-side sessions.

### 9.2. Redis как SPOF

Redis не должен быть единственной точкой отказа для критических сущностей.

Минимум:

- persistence (AOF/RDB).
- health checks.
- backup.

Критически важные сущности:

- session metadata
- plan metadata
- history/version index

Возможная схема:

- Redis — быстрый active layer.
- PostgreSQL — durable store для планов и истории.

### 9.3. PostgreSQL защита

На стороне БД:

- Prepared statements / parameterized queries.
- `statement_timeout` для защиты от runaway queries.
- Индексы на поля фильтрации и join-ключи.

## 10. Безопасность (упрощённо для хакатона)

### 10.1. Аутентификация и сессии

- Server-side session в Redis/PostgreSQL.
- `session_id` в `HttpOnly` cookie.
- TTL и idle timeout.
- Logout.

### 10.2. CSRF-защита

Так как используется cookie-based session, нужна CSRF-защита:

- `SameSite=Lax/Strict` для session cookie.
- Проверка `Origin/Referer` на backend.
- CSRF token по схеме double-submit cookie для state-changing запросов.

### 10.3. Rate limiting API

Нужны лимиты на уровне приложения:

- `/api/query`
- `/api/plan/update`
- `/api/export`
- `/api/login`

Механика:

- per-session/per-IP throttling
- circuit breaker на экспорт

### 10.4. Ввод пользователя и sanitization

System prompt не является защитой (и LLM нет), поэтому безопасность строится так:

- ограничение длины запроса
- trim/normalization
- удаление управляющих символов
- unicode normalization

Backend не доверяет пользовательскому вводу:

- все таблицы и поля проверяются по белым спискам
- все фильтры параметризуются
- обязательные ограничения добавляются на сервере

### 10.5. Логи

Минимум для прототипа:

- структурные логи (без лишних персональных данных)
- retention как конфиг

## 11. Производительность и UX таблицы

### 11.1. Пагинация

Лимит в десятки тысяч строк нельзя отдавать в DOM целиком.

Поэтому:

- серверная пагинация
- virtual scroll/windowing
- по умолчанию страница 100–500 строк
- сортировка и фильтрация выполняются на сервере

### 11.2. Экспорт

Экспорт Excel/CSV:

- асинхронно через job
- статус задачи
- большие файлы — потоково или временная ссылка

### 11.3. Ограничения тяжёлых запросов

На уровне DSL:

- лимит глубины `filter_tree`
- ограничение широких join
- hard limit на время выполнения и размер результата

## 12. Голосовой ввод

### 12.1. Цель

Дать пользователю возможность диктовать запрос в строку поиска/запроса.

### 12.2. Реализация

- Web Speech API (SpeechRecognition) на фронтенде, если доступно.
- Распознанный текст вставляется в поле запроса и обрабатывается как обычный текст.
- Fallback: если API недоступен — только текстовый ввод.

### 12.3. UX

- кнопка «микрофон» рядом со строкой запроса
- статус: слушаю → распознаю → готово
- стоп/повтор

## 13. План для хакатона

Чтобы успеть в срок, обязательный минимум:

- Server-side sessions.
- CSRF protection.
- DSL с `joins`, `filter_tree`, `plan_version`.
- Safe `calculate` без строковых формул.
- Server-side pagination + Excel-like UI с управлением колонками.
- ETL pipeline хотя бы в batch-виде + `dataset_version`.
- Семантический анализ запроса (словарь/индекс) + degraded mode.
- Rate limiting на `/api/query` и `/api/export`.
- История минимум на 10 версий плана.
- Redis persistence + резервное хранение планов в PostgreSQL.

## 14. Резюме архитектурных принципов

Система строится по принципу:

- Семантический слой интерпретирует намерение, но не управляет данными напрямую.
- Backend — единственный доверенный слой.
- Фронтенд не владеет состоянием плана.
- Join, фильтры, вычисления и экспорт выполняются только по белым спискам и серверным правилам.
- При сбое семантического слоя система не останавливается, а деградирует до ручного режима.
