# Семантический слой: словарь полей и связи (по CSV хакатона)

Этот документ фиксирует:

- **какие источники данных** используются (по папкам `/opt/hakaton/*`)
- **реальные поля (как в CSV)** и их нормализованные имена
- **ключи связки** и whitelist `relation` для `joins` (главный ключ — **КЦСР**)
- **что индексируем** для семантического поиска (значения/синонимы)

## 1) Источники (как в хакатоне)

### 1.1 `1. РЧБ/` (пример: `1. РЧБ/январь2025.csv`)

Файл содержит **преамбулу** (служебные строки), затем строку заголовков. ETL должен уметь находить заголовок по маркеру `Бюджет;Дата проводки;...`.

Заголовки (фрагмент):

- `Бюджет`
- `Дата проводки`
- `КФСР`, `Наименование КФСР`
- `КЦСР`, `Наименование КЦСР`
- `КВР`, `Наименование КВР`
- `КВСР`, `Наименование КВСР`
- `КОСГУ`, `Наименование КОСГУ`
- `Код цели`, `Наименование Код цели`
- `КВФО`, `Наименование КВФО`
- `Источник средств`
- `Лимиты ПБС 2025 год`
- `Подтв. лимитов по БО 2025 год`
- `Подтв. лимитов без БО 2025 год`
- `Остаток лимитов 2025 год`
- `Всего выбытий (бух.уч.)`

Ключи:

- **КЦСР**: строковый код вида `08.3.02.97070` (нужно нормализовать формат)
- `Дата проводки`: дата

### 1.2 `2. Соглашения/` (пример: `2. Соглашения/01012025-01042026.csv`)

Это структурированный CSV с заголовками в snake_case.

Ключевые поля:

- `close_date` (дата)
- `reg_number` / `main_reg_number`
- `amount_1year`
- `caption` (название бюджета/документа)
- `dd_recipient_caption` (получатель, может быть пустым)
- **коды**: `kfsr_code`, **`kcsr_code`**, `kvr_code`, `kesr_code`, `kadmr_code` и др.

Ключи:

- **`kcsr_code`**: строковый код (формат отличается от `РЧБ/`; нужна нормализация)

### 1.3 `3. ГЗ/` (госзаказ)

#### 1.3.1 `3. ГЗ/Бюджетные строки.csv`

Поля:

- `con_document_id` (идентификатор связки внутри ГЗ-части)
- `kfsr_code`, **`kcsr_code`**, `kvr_code`, `kesr_code`
- `kvsr_code`, `kdf_code`, `kde_code`, `kdr_code`, `kif_code`
- `purposefulgrant`

Ключи:

- `con_document_id` (связывает бюджетные строки ↔ контракт ↔ платежки)
- **`kcsr_code`** (для склейки с другими источниками по КЦСР)

#### 1.3.2 `3. ГЗ/Контракты и договора.csv`

Поля:

- `con_document_id`
- `con_number`
- `con_date`
- `con_amount`
- `zakazchik_key`

Ключи:

- `con_document_id`

#### 1.3.3 `3. ГЗ/Платежки.csv`

Поля:

- `con_document_id`
- `platezhka_paydate`
- `platezhka_key`
- `platezhka_num`
- `platezhka_amount`

Ключи:

- `con_document_id`

### 1.4 `4. Выгрузка БУАУ/` (пример: `4. Выгрузка БУАУ/хакатон БУАУ август 2025.csv`)

Заголовки (фрагмент):

- `Бюджет`
- `Дата проводки`
- `КФСР`
- **`КЦСР`**
- `КВР`
- `КОСГУ`
- `Код субсидии`
- `Отраслевой код`
- `КВФО`
- `Организация`
- `Орган, предоставляющий субсидии`
- `Выплаты с учетом возврата`
- `Выплаты - Исполнение`
- `Выплаты - Восстановление выплат - год`

Примечание: встречается строка `Итого...` — ETL должен уметь фильтровать агрегатные строки.

## 2) Нормализация ключей и полей (канонические имена)

Семантический слой работает по **каноническим полям**, а ETL маппит к ним сырьё.

### 2.1 КЦСР (главный join-ключ)

В разных источниках КЦСР представлен как:

- `КЦСР` (РЧБ/БУАУ) — формат с точками: `08.3.02.97070`
- `kcsr_code` (Соглашения/ГЗ) — формат без точек или иной кодировки: `0520197002`, `101016105Б`

Правило ETL: завести **два поля**:

- `kcsr_raw` — как в файле
- `kcsr_norm` — нормализованный ключ для join

Стратегия нормализации для хакатона (практичная):

- удалить пробелы
- привести к upper
- заменить кириллические `АВЕКМНОРСТУХ` на латиницу (и/или хранить отдельно), чтобы `105Е` и `105E` не расходились
- дополнительно хранить `kcsr_digits` (только цифры) как fallback-ключ для «похожих» кодов

Важно: финальную формулу `kcsr_norm` уточняем после проверки пересечений между источниками (какой именно вид реально “склеивается”).

### 2.2 Канонические поля (минимальный набор)

- `budget_name` ← `Бюджет` / `caption`
- `org_name` ← `Организация` (БУАУ)
- `recipient_name` ← `dd_recipient_caption` (Соглашения) и/или (в будущем) справочник получателей
- `posting_date` ← `Дата проводки` (РЧБ/БУАУ)
- `period` ← вычисляемый период (месяц/квартал из `posting_date` или `period_of_date`)
- `kfsr_code`, `kvr_code`, `kosgu_code`, `purposefulgrant_code` ← из соответствующих полей
- `limit_amount` ← `Лимиты ПБС ...`
- `spend_amount` ← `Всего выбытий (бух.уч.)` (РЧБ) или `Выплаты - Исполнение` (БУАУ)
- `payments_amount` ← сумма `platezhka_amount` (ГЗ)
- `contract_amount` ← `con_amount` (ГЗ)

Производные метрики (не храним как вводимые пользователем значения, строим сервером):

- `execution_percent` = `spend_amount / limit_amount` (safe divide)

## 3) Whitelist relations (joins) для DSL

Семантический слой выбирает `relation` из белого списка; backend знает, какие колонки реально join-ятся.

### 3.1 Внутри `3. ГЗ/`

- `gz_budgetlines_to_contracts_by_con_document_id`
  - left: `gz_budget_lines.con_document_id`
  - right: `gz_contracts.con_document_id`
- `gz_contracts_to_payments_by_con_document_id`
  - left: `gz_contracts.con_document_id`
  - right: `gz_payments.con_document_id`

### 3.2 Между источниками по КЦСР

Базовый набор для хакатона:

- `rchb_to_buau_by_kcsr_norm`
  - left: `rchb.kcsr_norm`
  - right: `buau.kcsr_norm`
- `rchb_to_agreements_by_kcsr_norm`
  - left: `rchb.kcsr_norm`
  - right: `agreements.kcsr_norm`
- `rchb_to_gz_budgetlines_by_kcsr_norm`
  - left: `rchb.kcsr_norm`
  - right: `gz_budget_lines.kcsr_norm`

Опционально (если нужно показывать «контракты/платежи» рядом с РЧБ):

- `rchb_to_gz_contracts_via_kcsr`
  - реляция “через бюджетные строки”: `rchb → gz_budget_lines (kcsr_norm) → gz_contracts (con_document_id)`
- `rchb_to_gz_payments_via_kcsr`
  - аналогично, но с агрегацией платежей по `con_document_id`

## 4) Что индексировать для семантического поиска

### 4.1 Индекс значений (entity index)

Минимум:

- все уникальные `kcsr_raw/kcsr_norm` (+ `Наименование КЦСР`, если есть)
- `budget_name`
- `org_name`
- справочники `kfsr_code`, `kvr_code`, `kosgu_code`
- `purposefulgrant` / `purposefulgrant_code` (ГЗ)

### 4.2 Синонимы (field synonyms)

Примеры:

- “кцср”, “код кцср”, “кцср программы” → `kcsr_norm`
- “кфср” → `kfsr_code`
- “квр” → `kvr_code`
- “косгу” → `kosgu_code`
- “лимиты”, “лимит”, “ЛБО” → `limit_amount`
- “расход”, “выбытия”, “исполнение”, “выплаты” → `spend_amount`
- “платежи”, “платежки” → `payments_amount`
- “контракт”, “договор” → `contract_amount`
- “освоение”, “процент освоения” → `execution_percent`

## 5) Как семантический слой собирает колонки «как просили»

Правило UX:

1. если пользователь явно назвал поля (“покажи КЦСР, бюджет, выплаты”) → именно они попадают в `columns`
2. если пользователь не назвал колонки → подставляется пресет “по умолчанию” (например: `kcsr`, `budget_name`, `posting_date`, `spend_amount`, `execution_percent`)
3. UI даёт список доступных колонок (из метаданных) для **добавить/убрать**, как в Excel
